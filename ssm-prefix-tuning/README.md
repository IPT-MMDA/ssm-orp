# Prefix Tuning State Space Model — Звіт

**Модель:** Mamba-130M · **Задача:** SST-2

---

## Мета

Дослідити, чи достатньо навчити невеликий набір "префіксних токенів" на вході замороженої SSM-моделі, щоб спрямувати її на нову задачу класифікації — без зміни жодного внутрішнього параметра моделі.

---

## Що було зроблено

Взяв попередньо навчену модель Mamba-130M і заморозив всі її ваги. До кожного вхідного тексту додав K навчальних векторів (префіксів) на початку. Під час навчання оновлював лише ці вектори та лінійний класифікаційний шар.

Для порівняння навчав три варіанти дообучення на одних і тих самих даних:

| Метод | Що оновлюється | Кількість параметрів |
|-------|---------------|----------------------|
| Повне дообучення | Всі ваги моделі | 129 мільйонів |
| LoRA | Малорангові матриці-адаптери | ~1.5 мільйона |
| Prefix Tuning | Лише префікси + класифікатор | ~9 тисяч |

---

## Результати порівняння методів

| Метод | Точність | 95% ДІ |
|-------|---------|--------|
| LoRA (r=8) | 66.9% | [63.9%, 70.0%] |
| Повне дообучення | 63.9% | [60.9%, 67.2%] |
| Prefix Tuning (K=10) | 50.0% | [46.7%, 53.3%] |
| Prefix + повторна ін'єкція | 50.0% | [46.7%, 53.3%] |

50% — рівень випадкового вгадування для бінарної задачі. Обидва варіанти prefix tuning не змогли навчитись.

## Вплив довжини префіксу K

| K | Точність | 95% ДІ |
|---|---------|--------|
| 1 | 52.3% | [49.0%, 55.3%] |
| 5 | 53.8% | [50.6%, 57.1%] |
| 10 | 55.5% | [52.1%, 58.7%] |
| 20 | 53.3% | [50.1%, 56.7%] |
| 50 | 50.9% | [47.5%, 53.9%] |

Збільшення K дає мінімальний і нестабільний ефект — всі значення залишаються поблизу випадкового вгадування. Очікуваної немонотонної кривої (зростання → пік → зниження) не спостерігається: модель ніколи не виходить зі стану стохастичного вгадування, щоб ефект "забування" через великі K міг проявитись.

## Проблема "згасаючої пам'яті"

**Проблема наявна**

У Transformer-архітектурі кожен токен може напряму "подивитись" на будь-який інший токен, включно з префіксними. Саме тому prefix tuning добре працює в трансформерах.

В SSM інформація передається через прихований стан `h`, що оновлюється покроково:

```
h_t = A · h_{t-1} + B · x_t
```

Матриця `A` має власні значення менше 1, тому вклад префіксу зменшується з кожним кроком. До кінця речення сигнал від початкового префіксу помножено на `A^T` — число, що прямує до нуля. Для послідовностей довжиною 64 токени та 24 шарів Mamba сигнал фактично зникає перед тим, як досягає класифікатора.

Це підтверджується числами: функція втрат prefix tuning майже не рухається впродовж трьох епох (0.813 → 0.748 → 0.768) — модель не здатна навчитись через відсутність корисного градієнтного сигналу.

---

## Висновок

Prefix tuning у тому вигляді, в якому він застосовується до Transformer-архітектур, не є достатнім для керування замороженою SSM-моделлю. Проблема "згасаючої пам'яті" повністю нівелює вплив префіксу на вихід моделі.

LoRA показала найкращий результат (66.9%), обійшовши навіть повне дообучення за 3 епохи — завдяки вбудованій регуляризації та тому, що вона змінює внутрішні обчислення моделі, а не лише вхідну послідовність.

---

## Примітка про умови експерименту

Усі експерименти проводились не дуже потужному GPU тому:

- навчання тривало лише 3 епохи (замість типових 5–10)
- максимальна довжина послідовності — 64 токени (замість 128)
- розмір батчу — 4 (замість 16–32)
- використовувались не всі тренувальні дані (підвибірка для пришвидшення)

Результати LoRA та повного дообучення (63–67%) є нижчими за типові значення для цієї моделі на повному датасеті (~90%). Проте відносна картина між методами є коректною і демонструє стабільну тенденцію: LoRA > повне дообучення >> prefix tuning. Результати є демонстраційними, а не виробничими.

## Структура проєкту

```
ssm-prefix-tuning/
├── README.md                       ← цей файл (звіт)
├── src/ssm_prefix_tuning/          ← основний код
│   ├── config.py                   # гіперпараметри
│   ├── prefix_encoder.py           # PrefixEncoder + PeriodicPrefixInjector
│   ├── model_wrapper.py            # MambaPrefixModel
│   ├── lora_model.py               # LoRA через PEFT
│   ├── data.py                     # завантаження SST-2
│   ├── trainer.py                  # єдиний цикл навчання
│   └── evaluator.py                # bootstrap CI, таблиці, графіки
├── experiments/
│   ├── run_prefix_tuning.py        # запуск prefix tuning
│   ├── run_lora.py                 # запуск LoRA
│   ├── run_full_finetune.py        # запуск повного дообучення
│   ├── run_prefix_length_sweep.py  # sweep по K
│   ├── run_comparison.py           # порівняння всіх методів
│   └── analyze.py                  # аналіз чекпоінтів + генерація графіків
├── tests/                          ← 56 юніт-тестів (без GPU)
├── results/
│   ├── comparison/                 # comparison.csv + epoch_results.json
│   ├── prefix_sweep/               # sweep_results.csv
│   └── figures/                    # графіки PNG
└── pyproject.toml                  ← залежності
```

### Запуск

```bash
# встановити залежності
poetry install

# юніт-тести
poetry run pytest tests/ -v -m "not slow"

# порівняння всіх методів (потрібен GPU)
poetry run python experiments/run_comparison.py \
    --device cuda --epochs 3 --batch-size 4 --max-seq-len 64 \
    --output-dir results/comparison

# sweep по довжині префіксу K (потрібен GPU)
poetry run python experiments/run_prefix_length_sweep.py \
    --device cuda --epochs 3 --batch-size 4 --max-seq-len 64 \
    --ks 1 5 10 20 50 --output-dir results/prefix_sweep

# аналіз збережених чекпоінтів та генерація графіків
poetry run python experiments/analyze.py --device cuda
# або лише графіки, якщо чекпоінти вже оцінені:
poetry run python experiments/analyze.py --skip-eval
```
